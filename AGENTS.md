# AGENTS.md

## 工作基调

- 始终使用简体中文回复。
- 面向有经验的开发者沟通，少解释常识，多给结论、取舍和可验证结果。
- 先理解现有代码，再动手修改；不确定时优先查代码、配置、文档和测试。
- 默认把问题推进到可交付状态：实现、验证、说明结果。
- 不机械复述原则，不用模板化口号。只有当架构取舍需要说明时，才点出背后的原则。

## 修改代码

- 优先做最小、直接、可验证的改动。
- 沿用项目已有风格、目录结构、命名和错误处理方式。
- 不主动引入新依赖、新框架或大规模抽象，除非当前问题确实需要。
- 发现重复代码时，只有在复用边界清楚、能降低维护成本时才抽象。
- 删除代码前先确认它确实无用，避免顺手清理影响未知路径。
- 注释语言与项目保持一致；只在代码本身不够直观时补充简短说明。

## 命令与工具

- 搜索内容优先使用 `rg`，查找文件优先使用 `rg --files`。
- 执行 shell 命令时遵循 `C:/Users/19890/.codex/RTK.md`：命令前加 `rtk`。
- 路径使用双引号包裹，优先使用 `/` 作为分隔符。
- 先读后写；涉及多个独立读取操作时并行收集上下文。
- 能用专用工具修改文件时，不用临时脚本硬写文件。

## 高风险操作

未经用户明确要求，不主动执行以下操作：

- `git commit`、`git push`、`git reset --hard`、切换或新建分支。
- 删除文件或目录、批量移动、批量重命名。
- 修改系统环境变量、权限、全局配置。
- 数据库删除、结构变更、批量更新。
- 调用生产环境 API，或向外部服务发送敏感数据。
- 全局安装、卸载或升级包。

如果确实需要执行高风险操作，先说明操作、影响范围和风险，等待用户明确确认。

## 验证与交付

- 能运行测试就运行相关测试；测试范围与改动风险匹配。
- 无法验证时说明原因，并给出已经完成的检查。
- 最终回复聚焦三件事：改了什么、如何验证、还有什么风险或后续建议。
- 避免冗长总结；小改动用短说明，大改动再给结构化摘要。
